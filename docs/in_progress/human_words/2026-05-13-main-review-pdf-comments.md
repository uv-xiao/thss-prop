# main-review.pdf 批注提取与规则化记录

来源：`report/main-review.pdf`，批注作者为肖有为，批注时间对应本轮修改前的 PDF。以下记录保留用户在 PDF 中的原始修改意图，并将可推广意见整理为后续写作规则来源。

## 可推广到全文的规则

- “不是并列关系，是包含关系”：应用压力、架构多元化、生产力和软件适配不是简单并列，应写成应用需求推动架构演进，架构多元化和规模化进一步放大生产力与软件适配问题。
- “我们应该在全文中去掉这部分的内容，包括综述和未来工作的对应章节。”：删除不属于报告主题的内容，不把无关生态或泛化主题纳入综述和未来工作。
- “这个说法应该避免，直接讲内容，不需要 围绕‘xxx’”：正文避免“本报告围绕……”等设计文档式过程表述。
- “无需罗列，用‘形式化方法与多层抽象机制’概括即可”：摘要和总述中不要堆叠技术名词，优先用概括性学术表述。
- “删掉”：摘要页关键词和英文摘要页关键词需要删除；英文摘要页删除题名、作者等冗余块，只保留 Abstract。
- “无需英文缩写”：中文正文中普通术语优先中文表达，人工智能、大语言模型等不需要反复括注英文缩写。
- “引用位置应该放在‘OpenAI 对 Codex 的系统说明’后，而不是句末。”、“不要把引用放在句末”：引用应紧跟被讨论的工作名或材料名。
- “减少这种否定性的表达。”、“不要说‘只有..., 才可能...’, 正向的讨论机遇和挑战。”：减少否定式和条件式压迫表达，改用正向学术叙述。
- “不用说 本报告所说的xxx，直接说敏捷芯片设计即可”：避免定义前缀和元话语。
- “减少句中的‘’”、“不要这种行内‘’”：减少行内引号，除专名、引用或确需强调的术语外直接写入句子。
- “首次出现需要提供英文（E-graph）”：等价图首次出现需括注英文，后文中文优先。
- “使用FPGA来简写，首次出现需要提供中英文的全称”：现场可编程门阵列首次出现给出英文全称和 FPGA，后文可用 FPGA。
- “这个工作叫egglog”：Datalog 与等价饱和结合的相关工作应写作 egglog。
- “[74][75]是一篇工作”：ENumo 相关引用需避免拆成两篇工作处理。
- “应该概括为等价饱和在芯片设计EDA流程中的应用。”：硬件相关等价饱和段落应归为芯片设计自动化流程中的应用。
- “这一段落可以去掉”：PyTorch/TorchDynamo/TensorRT-LLM 的材料不应突兀形成独立段落，应融入架构约束下的编译优化逻辑。
- “还需要添加仓库级硬件设计：https://arxiv.org/abs/2604.14709，以及专门做RTL生成的：https://arxiv.org/abs/2506.14074”：智能体方法综述需补入仓库级硬件缺陷修复和寄存器传输级生成/验证基准。
- “不需要构成主逻辑链条这种表达，直接去描述和解释逻辑链条。”：避免说“构成主逻辑链条”，直接解释问题间的递进关系。
- “描述为构建面向智能体的架构-硬件-编译器协同链路。主要拆解为两个工作：架构芯片协同生成，以及智能体编译基础设施与工作流。”：未来工作总述要突出面向智能体的架构-硬件-编译器协同链路，并显式拆解架构芯片协同生成、智能体编译基础设施与工作流。
