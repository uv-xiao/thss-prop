# 开题报告整体结构重构 Spec

## 目的

本文件定义下一轮报告重构的目标结构。后续实施应先围绕本 spec 讨论和确认，再重写 `report/main.typ`，不要在当前草稿上继续局部增量修补。

参考对象：`resources/博士开题报告-陈仁泽.pdf`。学习其章节组织、引言的综述式铺陈、已有工作分章展开、未来工作单章规划和结论收束方式；不复制其措辞。

## 顶层主题

报告主题保持为：

> 敏捷芯片设计与创新性编译技术驱动的软硬件协同

主题解释：

- “敏捷芯片设计”强调面向领域需求快速形成、验证和迭代硬件能力。
- “创新性编译技术”不是辅助工具，而是连接应用需求、架构抽象、硬件结构、综合优化、验证证据和系统软件的重要接口。
- “软硬件协同”不是泛泛地同时设计软件和硬件，而是面向跨硬件架构的协同生态，围绕应用模式发现、架构/硬件能力生成、编译映射、综合优化、验证约束体系和智能体能力提升构建系统方法。

## 与当前草稿的关键差异

当前草稿的问题：

- 引言仍偏“研究背景 + 研究目标”的概括写法，没有形成类似范文第一章的综述式总分结构。
- 国内外现状与挑战虽然已扩写，但应被吸收到第一章引言中，作为总论证的一部分，而不是单独作为一个短章。
- 技术部分仍以“已有研究基础”统摄多个方向，无法给代表性工作足够章节空间。
- EggMind 当前被放在未来工作中，但它应作为现有工作的一部分呈现。
- 未来工作需要单独成章，围绕尚未完成的三个概括性研究方向展开，而不是直接用工具或仓库名组织，也不能混入已有工作叙述。

重构后的原则：

- 第一章是完整综述式引言，承担背景、趋势、研究归类、问题挑战、研究目标和章节安排。
- 技术部分按现有工作的几个层次分别成章，每章形成“问题-现状-方法-系统-评估-小结”的闭环。
- EggMind 放入现有工作章节，作为 e-graph 策略智能体化和 agentic 编译优化积累。
- 最后一个主要技术章介绍未来工作，采用“研究背景与问题分析、框架设计思路、关键技术实现、预期技术特征、验证与应用规划”的范文式结构。

## 建议全文结构

### 摘要

摘要需要在较短篇幅内完成四件事：

1. 说明算力需求、领域定制和芯片设计生产力的矛盾。
2. 提出“敏捷芯片设计与创新性编译技术驱动的软硬件协同”主题。
3. 概括已有工作三大层次。
4. 概括未来工作目标与预期贡献。

摘要不要写成论文成果列表。每个成果只在说明主线时出现。

### 第一章 引言

第一章应是综述形式的总分结构，篇幅应显著超过当前草稿的引言。它需要像范文一样，从技术背景逐层收束到研究问题。

第一章章标题固定为“引言”。不要改成“研究背景与问题提出”或更长的说明性标题；具体论证功能通过 1.1-1.6 的小节标题和两张总览图体现。第一章不单独设置“报告结构安排”小节。

第一章前半部分不要过早介绍本人工作。1.1-1.4 应先完整铺陈产业趋势、技术路线、协同接口分类、研究现状与未解决问题；本人已有工作和未来计划集中在 1.6 “已有研究基础与拟开展研究”中出现。1.4 可以说明各类方法仍存在的问题，但不要插入“本人工作解决了 X”的成果展开。

第一章需要更多业界数据、顶级体系结构会议论文和可核验模型来支撑结论。写作前应参考 `docs/writing/introduction-evidence-plan.md`，优先使用大公司作者机构、权威专家、工业系统论文、官方技术报告，以及 ISCA/MICRO/HPCA/ASPLOS 近年论文、CACM/Nature 等权威来源中的数据。每个数据点必须说明指标、基线、工作负载和适用范围。支撑异构架构复杂度与编程关切时，优先考虑 ATLAS/DeepStack、Huawei CloudMatrix384/UB-Mesh、NVIDIA GB200 NVL72/IMEX 等材料，不把 ISCA AIO 或 MICRO SCAR 作为主要证据。支撑“架构约束下的编译优化”接口时，优先使用 PyTorch 2 `torch.compile`/TorchDynamo、NVIDIA TensorRT-LLM 等真实编译框架和推理系统；硬件系统材料用于说明该接口为什么在工业异构架构上更复杂。支撑 LLM/agentic 技术趋势时，优先使用 NVIDIA AVO、Google DeepMind AlphaEvolve、Anthropic/OpenAI 官方技术报告或系统说明。

建议章节：

#### 1.1 领域应用演进与领域定制计算趋势

功能：

- 建立产业和技术背景。
- 以 AI/LLM 为主线建立最强产业压力，再扩展到后量子密码、信号处理、点云处理、图形渲染等场景，说明领域定制需求不是 LLM 单点，而是由不同计算、访存和控制模式共同驱动。
- 把 LLM/agentic 方法作为技术革新和行业趋势引入，但要区分两层含义：LLM workload 推高算力和领域定制需求；LLM/agentic workflow 正在改变复杂工程系统的自动化方式。
- 说明先进工艺收益放缓、通用处理器能效限制和应用迭代速度共同推动领域定制。

写作要求：

- 不要只说“算力紧缺很重要”。
- 要解释“需求增长”和“设计生产力不足”是共同瓶颈。
- 应用场景比例采用“AI/LLM 为主，其他领域作为多样性扩展”。不要只写 LLM，也不要均匀铺开所有场景。
- LLM/agentic 趋势在本节只需作为产业和工程自动化趋势点出，不展开方法细节。优先使用 Anthropic/OpenAI 官方 agentic coding 或 agent 系统材料，也可用 McKinsey 等企业报告说明 agentic AI 正在进入软件工程和企业工程流程；必须标明这只是趋势证据，不是芯片设计技术效果证据。
- 可引用 APS/Aquas/ISAMORE/SkyEgg 等 introduction 中对应用领域的描述。
- 需要使用外部数据或工业系统论文支撑趋势判断，例如 DSA、TPU、AI/ML 工作负载、异构加速器或芯片设计生产力相关论文。
- 证据进入正文优先采用“关键转折处代表性例子”的方式。证据表格不是硬性要求；只有在不破坏引言学术叙事流畅性的情况下，才加入简短的“代表性工业系统与研究证据”表。

#### 1.2 芯片设计生产力瓶颈与敏捷设计路径

功能：

- 先讲芯片设计生产力和迭代速度瓶颈，再引出“敏捷芯片设计”术语。
- 综述敏捷芯片设计的主要技术路线。
- 包括 ASIP/ISAX、RISC-V 扩展接口、FPGA/可重构加速器、HLS/硬件前端、AI 辅助硬件设计。
- 将 agentic/LLM 方法作为设计自动化的新兴路线之一介绍，说明其行业动因是长周期、多工具、专家密集的芯片设计流程需要更强的自动化与协同编排。

写作要求：

- 每类路线都要说明其价值和局限。
- 这里是综述，不介绍个人工作细节。
- 结尾需要指出：硬件创新若缺少软件、编译和验证连接，难以形成可复用系统能力。
- 应结合顶级体系结构会议近年论文说明：早期架构分析、设计空间探索、可重定向编译支持和自动化设计工具为什么成为生产力问题。
- “敏捷芯片设计”的定义应在本节中段自然出现：它不是单纯缩短 RTL 编写时间，而是快速形成、验证、映射和迭代领域硬件能力。
- 使用 NVIDIA AVO、Google AlphaEvolve、OpenAI Codex/Operator/ChatGPT agent system cards、Anthropic Agentic Coding Trends Report、Siemens Fuse EDA AI Agent、Synopsys AgentEngineer、Agentic EDA survey、verification-agent 论文等材料时，应分清证据功能：AVO/AlphaEvolve 支撑“agent + evaluator/harness + execution feedback”的技术机制；Anthropic/OpenAI 支撑 agentic coding、tool use、sandbox/test/review/human oversight/safety boundary；Siemens/Synopsys 支撑 EDA 行业趋势。必须同时指出芯片设计中的 correctness、verification、maintainability、observability 风险，使 1.3 的 harness 论证自然出现。

#### 1.3 跨硬件架构的软硬件协同生态与关键接口

功能：

- 明确本报告不是单一硬件工具或单一编译优化集合，而是面向跨硬件架构的软硬件协同生态。
- 说明编译与架构两个层次是衔接软件需求和硬件实现的重要接口。
- 从架构抽象、编译映射、程序等价空间、硬件综合 IR、e-graph、验证证据、智能体能力提升几个角度解释协同生态如何形成。
- 补充“编译基础设施作为 agent harness 介质”的论证：编译基础设施不只是后端工程，而是把目标事实、IR、语义契约、受约束变换、可执行后端产物和证据反馈组织起来的中间层。该论证可参考 `resources/compiler-infra-is-harness-medium.md`，但正文应转化为开题报告的学术叙事。
- 明确本报告对 LLM/agentic 的定位：它是软硬件协同自动化的重要技术变量，但只有在架构接口、编译基础设施、语义 gate、性能剖析 feedback 和 evidence memory 约束下，才可能可靠服务跨硬件架构协同。

写作要求：

- 不能把编译写成后端工具。
- 不能把架构写成固定 ISA 或单一硬件实例，而应写成连接领域需求、软件接口、硬件能力和工具链约束的抽象层。
- 必须明确区分两个关键接口：
  - 架构接口：把应用需求转化为硬件可提供的能力边界，包括 ISA/ISAX、微架构机制、存储层次、互连、加速器执行模型、软件可见资源抽象等。
  - 编译接口：把软件语义转化为可映射、可优化、可验证的执行形式，包括 IR、lowering、pattern/rewrite、调度、代码生成、runtime API、profiling feedback 等；在 agentic 方法中，它还应显式提供 compiler-native objects、pass/action 边界、target contracts、semantic/executable checks、benchmark/trace evidence，避免退化为自由文本或自由代码生成。
- 二者关系应写清楚：架构接口定义“硬件能提供什么能力”，编译接口定义“软件如何发现、使用、优化和验证这些能力”，智能体能力提升“这些能力在工业系统中如何被调度和演化”。
- 要突出编译与架构共同承担三类职责：
  - 发现：从应用中发现值得硬化或优化的模式；
  - 连接：把应用语义、架构接口、硬件实现、综合流程和系统执行约束连接起来；
  - 证明：产生可检查、可验证、可复现的证据。
- 本节应避免把 agent 写成额外装饰。若提到 agent，应强调 agent 只有在编译基础设施提供结构化对象、语义边界和证据门控时才适合进入协同流程。
- 1.3 末尾或 1.5 开头需要有一个“中心论断”句，作为从背景综述过渡到三个核心问题的腰部命题。中心论断应表达：跨硬件架构软硬件协同的关键不在单次生成硬件模块或单次优化程序，而在通过架构接口定义可复用硬件能力，通过编译接口连接软件语义、硬件实现与智能体能力提升，并在形式化证据和智能体化工具链支撑下形成可持续演化的智能体能力支撑体系。具体措辞可在正文写作时润色，不必机械照抄。

图示要求：

- 本节附近需要新绘制一张“跨硬件架构软硬件协同生态”总览图。
- 图中应体现：应用需求、架构接口、编译接口、硬件实现、综合/验证、智能体能力提升之间的关系。
- 图的重点是解释编译与架构作为两个关键接口，而不是展示某个具体工具。

#### 1.4 面向协同接口的国内外研究现状

功能：

对相关研究做分类综述，为后文三个现有工作章建立评价坐标。本节应以“协同接口”为主线，而不是按论文或技术对象机械分类。技术对象可以作为每类接口内部的代表路线出现。

建议分为：

1. 应用到架构：领域需求如何转化为架构、指令和微架构能力。内部可讨论 ASIP/ISAX、RISC-V extension、领域定制处理器、自定义指令识别与复用。
2. 架构到硬件：架构能力如何转化为可综合、可验证、质量可控的硬件实现。内部可讨论硬件前端、HLS、硬件 IR、综合优化和微架构表达。
3. 架构约束下的编译优化：程序语义如何进入可优化、可证明、可迁移的中间表示和优化空间，以及编译优化如何受到目标架构、异构执行形态、数据移动、系统软件边界和性能剖析的影响。内部可讨论 MLIR、e-graph、equality saturation、anti-unification、编译优化策略、TensorRT-LLM、PyTorch 2 `torch.compile`/TorchDynamo、NVL72/IMEX、CloudMatrix384/UB-Mesh 等材料。TensorRT-LLM 和 PyTorch/Dynamo 不应独立引出系统软件综述，而应作为真实编译优化框架和目标约束进入编译基础设施的证据。
4. 智能体化方法：大模型与形式化技术如何提升跨层协同的生成、搜索、验证和策略迁移能力。内部可讨论 AI 辅助 RTL/硬件设计、Agent Harness、形式化约束和可验证反馈。

在第 3、4 类之间需要加入一个清晰过渡：编译基础设施之所以能够支撑智能体化方法，不是因为 LLM 可以任意生成编译器，而是因为成熟编译系统已经提供了结构化 IR、pass 边界、目标契约、语义检查、执行轨迹和 profiling 反馈。正文可将 XLA/StableHLO/HLO/PJRT、Arknife、ACT、TL 等材料作为“目标事实如何进入编译基础设施”的代表性证据，用来支撑“target facts -> explicit IR and semantic contracts -> constrained transformations -> executable backend artifacts -> evidence and feedback”的论证链；将 NVIDIA AVO 与 Google AlphaEvolve 作为“agentic search/evolution 必须依靠 evaluator、execution feedback、domain knowledge 和可验证产物”的代表性证据；将 Anthropic/OpenAI 技术报告作为“agentic coding 需要 sandbox、tool use、test/review、人类监督和安全边界”的工程证据。这些例子应服务综述逻辑，不写成项目列表。

智能体化方法小类应形成递进链条，而不是平铺材料：先用传统 AI for EDA 简短说明 AI 已进入 floorplanning、DSE、verification 和 optimization 等点问题；再转向 agentic workflow，即模型能够读取上下文、调用工具、修改代码、运行测试并根据反馈迭代；随后用 OpenAI/Anthropic 说明真实工程工具环境中的 sandbox/test/review/human oversight/safety boundary，用 Google AlphaEvolve 说明 evaluator 约束下的非平凡算法发现，用 NVIDIA AVO 说明该模式已经进入工业 GPU kernel/compiler optimization 邻域，再用 Agentic EDA/verification-agent 材料说明芯片设计正在探索类似流程但风险更高。该小类结尾应显式判断：agentic 方法的核心并不只是更强的大模型，而是模型、工具、目标约束、执行反馈与证据记忆构成的闭环；对于芯片设计与编译优化而言，这一闭环必须进一步嵌入架构接口、编译 IR、形式化语义、性能剖析 和硬件验证流程之中。

上述顺序固定。它表达从应用需求进入系统，到架构能力定义、硬件实现，再到受架构约束的编译优化和智能体化方法的叙事顺序。不要把编译优化综述提前到“架构到硬件”之前。

写作要求：

- 这部分应系统整理 `resources/` 内各论文的 related work，不凭泛泛记忆写。
- 这部分还应补充外部顶会论文和业界系统数据，尤其用于支撑四类综述问题的成熟点和未解决问题。
- 每一类都要有“已有路线 -> 成熟点 -> 未解决问题”的结构。
- 不要把个人工作提前作为解决方案展开，只可在小结中引出“本人已有工作围绕这些问题展开”。
- 每类标题必须体现“接口问题”，而不是只列技术名。
- 证据表格最多作为可选短表；如果会打断 1.4 到 1.5 的问题收束，则不放入正文。

#### 1.5 核心科学问题与研究挑战

功能：

把 1.1-1.4 的综述收束为本报告研究问题。

本节必须收束为三个核心研究问题。1.4 中的四类综述问题不能继续平铺到 1.5，而应合并回这三个问题，形成后文三章现有工作和一章未来工作的总纲。

三个核心研究问题：

1. 应用需求如何通过架构接口转化为可复用的指令、微架构和加速器能力；
2. 架构能力如何高质量落到可综合、可验证、性能与资源可控的硬件实现；
3. 编译接口如何在架构约束下组织程序语义、硬件能力和优化过程，形成可验证、可复用、可被智能体使用的编译协同机制。

写作要求：

- 挑战要具体，避免“研究 X 方法”式空话。
- 每个挑战需要指向后续一个技术章或未来工作章。
- 第三个问题应吸收“编译基础设施作为 harness 介质”的思想：问题不只是如何生成一个 pass 或 backend，而是如何把程序语义、目标事实、变换边界、验证义务、执行证据和性能反馈组织成可检查、可复用、可演化的验证约束体系。
- 不要拆成四个或五个核心问题；四类综述问题是综述分类，三个问题是研究问题收束。
- 本节不要使用表格列出核心问题、接口、章节和代表工作。三问题应通过连续论证自然收束，保持博士开题报告的学术叙事流畅性。结构映射交给 1.6 末尾的文字和图示完成。

#### 1.6 已有研究基础与拟开展研究

功能：

#### 5.4 面向工业级异构架构的算子优化与智能体化编译协同

按同样五段结构。

本方向标题采用“面向工业级异构架构的算子优化与智能体化编译协同”。相比“面向异构任务图的编译-智能体能力提升”或“算子-智能体化编译协同”，该表述更准确：它明确呈现两条并列子线，即工业级异构架构上的芯片算子优化，以及系统软件与编译技术的协同。`resources/seed-proposal.docx` 的内容应作为本节未来工作的重要来源，尤其用于支撑“面向新领域、长任务的芯片算子自动优化 Agent”这一子方向。

本方向内部采用并列双子线组织，而不是上下游闭环绑定：

1. 面向工业级异构架构的芯片算子自动优化，重点来自 `resources/seed-proposal.docx`，包括复杂 fused 算子、Torch 训练/推理算子、长程 Agent Harness、IR/Pass/profiling/形式化辅助优化和回归验证。
2. 面向工业级异构架构的分布式任务执行协同，重点来自 PTO Runtime distributed features，包括任务图执行、异构设备边界、数据移动、ready/completion protocol、TensorMap/RingBuffer 扩展和可观测性。

二者的汇合点是工业级异构架构，而不是彼此承担直接上下游关系。写作时可以说明它们共同服务算子与智能体能力提升，但不要把 PTO Runtime distributed features 写成 Agent Harness 的部署后端，也不要把 Agent Harness 写成 PTO Runtime 的上层计划。

重点：

- 可以在本节中说明 PTO Runtime distributed features 作为该方向的已有基础或原型，但小节论证对象是“面向工业级异构架构的算子优化与智能体化编译协同”。
- 可以在本节中承接 `resources/seed-proposal.docx`，但正文要抽象成研究问题，不写成公司项目申请书。
- Agent Harness 只在需要解释长程芯片算子自动优化时出现，不能覆盖整个 PTO Runtime distributed features 原型。PTO Runtime distributed features 暂时没有 agent 计划，应作为智能体能力提升、任务图执行、数据移动和可观测性的已有基础或系统原型来写。
- 写 AVO 时应把它作为“方向正确但问题仍未被系统解决”的外部证据，而不是窄义 baseline 或竞品。AVO 说明在 Blackwell B200 等工业 GPU 上，agentic evolutionary search 可以在 attention kernel 层面形成可测性能收益；但本方向研究对象更宽，包括复杂 fused 算子、Torch 训练/推理核心算子、跨算子流水和智能体能力提升，约束也更系统，包括 IR/Pass 边界、形式化约束、profiling、回归验证、系统编排证据和长期可维护的经验记忆。
- 研究对象：工业级异构架构上的复杂 fused 算子、Torch 训练/推理核心算子、算子生成与改写、系统编排、性能反馈和回归验证。
- 核心问题：新领域算子优化依赖架构知识、ISA/存储层次、编译约束和专家经验，普通代码生成 Agent 难以长期稳定地完成正确性保持、性能调优和跨算子协同优化。
- 方法方向：构建长程 Agent Harness，提供架构描述、算子需求、IR/Pass 生成边界、工具调用、自动验证、性能 profiling、经验记忆和回归分析。
- 形式化与编译支撑：结合 IR 语义建模、优化 Pass 自动生成、profiling 反馈和形式化建模/求解，扩大优化空间并控制搜索复杂度。
- 预期验证：以 correctness/performance 为评价函数，覆盖核心 fused 算子、Torch 训练算子和跨算子流水编排，在真实训练或推理任务中评估性能、稳定性和可维护性。
- task dependency graph；
- host/AICPU/AICore 或异构设备边界；
- distributed task graph；
- ready/completion protocol；
- TensorMap/RingBuffer 扩展；
- 性能剖析、执行轨迹与编译决策的相互校验。
