# 开题报告 Agent 撰写规范

## 目标

本项目的目标是撰写一份中文为主、术语可保留英文的博士开题报告材料。报告必须围绕同一条主线展开：以“敏捷芯片设计与创新性编译技术驱动的软硬件协同”为主题，在算力紧缺和领域应用快速演进的背景下，通过敏捷芯片设计、可重定向编译、指令/微架构定制、硬件前端抽象、综合优化，以及大模型与形式化技术结合的自动化方法，面向跨硬件架构的软硬件协同生态提升系统生产力。其中，编译与架构是衔接软件需求和硬件实现的两个关键接口。

## 资料优先级

1. `resources/2201111592 肖有为 事迹材料.docx` 是最高优先级主线资料，负责确定叙事范围、贡献分类和未来发展方向。
2. `resources/` 下各论文仓库的 `README.md`、`main.tex`、`sec/*`、`doc/*` 是技术细节来源。
3. `resources/` 下现有论文正文、草稿、图表、表格和 BibTeX 足以支撑详尽写作。正式扩写任何代表性工作前，必须先阅读对应资源并提炼可验证材料。
4. 没有在仓库或事迹材料中出现的信息不得写成既定事实；可写入 `TODO` 或“待补充/待核验”。
5. 涉及会议、作者顺序、专利状态、获奖名次、项目编号等事实时，只能引用已给材料中的信息，不能凭印象补充。

## 总体写法

1. 采用“行业背景/现状 -> 核心挑战 -> 研究目标 -> 已有基础 -> 未来工作”的闭环。
2. 引言中的国内外研究现状按“协同接口”分类，而不是按论文或技术对象分类：应用到架构、架构到硬件、软件到编译、编译到运行时、大模型与形式化技术支撑的自动化方法。
3. 引言最后必须收束为三个核心研究问题，不要写成四个或五个。五类协同接口用于综述分类，三个问题用于支撑后文结构。
4. 引言末尾的章节映射采用“主问题 + 次级支撑”：每章明确一个主问题，同时说明其如何支撑其他问题；不要写成僵硬的一一对应，也不要写成散乱网络。
5. 第一章 1.1-1.4 不要提前展开本人工作；先完成产业趋势、技术路线、研究现状和挑战铺陈。本人已有工作与未来计划集中放在 1.6。
6. 第一章必须使用业界数据、顶级体系结构会议论文或权威系统论文支撑关键判断。写作前阅读 `docs/writing/introduction-evidence-plan.md`，并为每个数据点核验指标、基线、工作负载和适用范围。
7. 每个技术方向必须回答三个问题：
   - 为什么行业需要这个方向？
   - 现有方法卡在哪里？
   - 本人的工作如何构成可复用的方法、工具或系统能力？
8. 避免成果堆砌。每项工作都要放回主线中，说明它在“系统生产力”链条中的位置。
9. 第一作者代表性工作要详尽展开：问题背景、关键挑战、核心思想、系统设计、技术机制、实验设置、结果和意义都要写。
10. 其他工作概括介绍：强调它们如何补全技术链条，不展开实现细节。
11. 未来工作不能变成项目列表；必须从已有工作自然推出，说明未来要解决的剩余矛盾。
12. 第一层“领域定制与端到端协同”中，APS/Aquas 可以作为框架入口先写，但 ISAMORE 是同等重要的代表性工作，正文篇幅不得少于 APS/Aquas 的合计篇幅。
13. 协同框架和单点优化技术同样重要。不要为了强调系统性而压缩单点优化技术的技术深度。
14. 当前正文只能视为骨架。后续质量提升必须显著扩写各部分，形成耐心、严肃、完整的故事讲述。

## 语言风格

1. 中文为主，保留必要英文术语，如 MLIR、e-graph、equality saturation、anti-unification、ISAX、ASIP、RTL、HLS、LLM、TraceDB。
2. 中文句子要短，避免一段中连续堆叠三个以上抽象名词。
3. 技术术语第一次出现时给出中文解释或上下文。
4. 不使用宣传式表达，例如“国际领先”“填补空白”等，除非有明确证据。优先写“提出了”“实现了”“验证了”“支撑了”。
5. 结果数字必须带比较对象和场景，例如“相比 Vitis HLS 平均 3.01x、最高 5.22x”。

## 结构规范

建议正文结构：

1. 绪论：算力紧缺、领域定制、芯片设计生产力瓶颈。
2. 国内外研究现状与挑战：敏捷芯片设计、ASIP/ISAX、硬件设计前端、HLS/综合、AI for EDA。
3. 研究基础与已有工作：
   - 领域定制与端到端协同：APS/Aquas + ISAMORE + Cayman。
   - 硬件前端与综合优化：Cement、SkyEgg、Clay，并着重概括 HECTOR。
   - 大模型与形式化技术驱动的软硬件协同方法：EggMind、OriGen，并连接未来三个概括性研究方向。
4. 未来研究计划：
   - 可解释、可审计的编译基础设施。
   - 验证约束下的架构、硬件与编译协同生成。
   - 面向工业级异构架构的算子优化与运行时编译协同。
   - IntelliC、Spine、PTO Runtime distributed features 只能作为已有基础、原型或内部代号出现，不作为方向标题。
   - `resources/seed-proposal.docx` 的内容属于“面向工业级异构架构的算子优化与运行时编译协同”方向，写作时应抽象为研究问题和技术路线，不照搬项目申请书格式。
   - Agent Harness 只在长程芯片算子自动优化等需要它的子线中出现；不要把 PTO Runtime distributed features 写成 agent 项目。
   - 算子自动优化 Agent Harness 与 PTO Runtime distributed features 是并列子线，汇合点是工业级异构架构；不要强行写成上下游闭环。
5. 预期成果与计划安排。

## 图示规范

1. 正文需要适当添加图片，支撑总体逻辑、系统结构、方法流程和实验结果说明。
2. 已有系统图、流程图、实验图优先从 `resources/` 中选用，使用前必须确认图的来源、含义和与正文论点的对应关系。
3. 如果 `resources/` 中没有合适图片，例如整体研究主线、章节关系、未来工作闭环、跨层软硬件协同框架等，应直接绘制新图。
4. 第一章必须新绘制两张总览图：一张“跨硬件架构软硬件协同生态”概念图，一张“研究问题与章节映射”结构图。
5. 新图必须服务论证，不做装饰。优先使用框架图、流程图、分层图和证据闭环图。
6. 每张图都要在正文中被解释：图前说明它回答的问题，图后说明它支撑的论点。
7. 图题和图内文字中文为主，保留必要英文术语，并与术语表保持一致。

## 代表性工作展开模板

每个代表性工作按以下顺序写：

1. 背景：该问题对应哪类产业或学术瓶颈。
2. 挑战：至少两点，且要具体。
3. 方法：一句话概括核心洞见，再分解为系统组件。
4. 结果：性能、资源、开源、会议、落地等可验证指标。
5. 意义：它如何补强主线，而不只是单点优化。

ISAMORE 展开时必须同时详细回答：

1. 可复用自定义指令为什么是敏捷芯片设计中连接应用模式与硬件能力的关键机制。
2. e-graph anti-unification 为什么是发现可复用指令模式的核心方法创新。

## 迭代规则

1. 每次迭代先检查主线是否一致，再改局部文字。
2. 当新增成果时，先更新 `report_spec.md` 的成果矩阵，再更新正文。
3. 当无法确认事实时，在正文用 `TODO:` 标记，不把猜测写成定论。
4. 正式提交前至少完成三轮检查：
   - 逻辑闭环检查：背景、挑战、方案、未来是否呼应。
   - 事实检查：数字、会议、作者顺序、专利、获奖是否可追溯。
   - 语言检查：中文是否顺畅，术语是否统一，段落是否过长。
