# 开题报告 Agent 撰写规范

## 目标

本项目的目标是撰写一份中文为主、术语可保留英文的博士开题报告材料。报告必须围绕同一条主线展开：以“敏捷芯片设计与创新性编译技术驱动的软硬件协同”为主题，在算力紧缺和领域应用快速演进的背景下，通过敏捷芯片设计、可重定向编译、指令/微架构定制、硬件前端抽象、综合优化，以及大模型与形式化技术结合的智能体化方法，面向跨硬件架构的软硬件协同生态提升系统生产力。其中，编译与架构是衔接软件需求和硬件实现的两个关键接口。

## 资料优先级

1. `resources/2201111592 肖有为 事迹材料.docx` 是最高优先级主线资料，负责确定叙事范围、贡献分类和未来发展方向。
2. `resources/` 下各论文仓库的 `README.md`、`main.tex`、`sec/*`、`doc/*` 是技术细节来源。
3. `resources/` 下现有论文正文、草稿、图表、表格和 BibTeX 足以支撑详尽写作。正式扩写任何代表性工作前，必须先阅读对应资源并提炼可验证材料。
4. 没有在仓库或事迹材料中出现的信息不得写成既定事实；可写入 `TODO` 或“待补充/待核验”。
5. 涉及会议、作者顺序、专利状态、获奖名次、项目编号等事实时，只能引用已给材料中的信息，不能凭印象补充。

## 总体写法

1. 采用“行业背景/现状 -> 核心挑战 -> 研究目标 -> 已有基础 -> 未来工作”的闭环。
2. 引言中的国内外研究现状按“协同接口”分类，而不是按论文或技术对象分类：应用到架构、架构到硬件、架构约束下的编译优化、大模型与形式化技术支撑的智能体化方法。
3. 第一章 1.4 的综述顺序固定为：应用到架构、架构到硬件、架构约束下的编译优化、智能体化方法。原独立系统软件材料必须并入第三类，不得独立成节。
4. 引言最后必须收束为三个核心研究问题，不要写成四个或五个。四类综述用于组织相关工作，三个问题用于支撑后文结构。
5. 不要在 1.5 添加核心问题映射表。三问题通过连续论证自然收束，结构映射由 1.6 末尾的文字和图示承担。
6. 引言末尾的章节映射采用“主问题 + 次级支撑”：每章明确一个主问题，同时说明其如何支撑其他问题；不要写成僵硬的一一对应，也不要写成散乱网络。
7. 三个核心研究问题中，问题 1 应显式使用“架构接口”，问题 3 应显式使用“编译接口”。
8. 第一章章标题保持“引言”，具体性由小节标题和图示承担；不设置独立“报告结构安排”小节。
9. 第一章 1.1-1.4 不要提前展开本人工作；先完成产业趋势、技术路线、研究现状和挑战铺陈。本人已有工作与未来计划集中放在 1.6。
10. 第一章 1.1 的应用场景比例采用“AI/LLM 为主，其他领域作为多样性扩展”。PQC、DSP、点云、图形等场景用于说明领域定制需求的多样性，不要把报告写窄成 LLM serving。
11. “敏捷芯片设计”术语在第一章 1.2 中引入。先讲设计生产力和迭代瓶颈，再定义为快速形成、验证、映射和迭代领域硬件能力，而不是单纯缩短 RTL 编写时间。
12. 第一章需要把 LLM/agentic 方法作为技术革新和行业趋势介绍。1.1 短段落说明 LLM/agentic 正在改变复杂工程系统自动化范式；1.2 将其纳入芯片设计生产力和智能体化路线；1.3 再给出约束性判断：agentic 方法必须依托架构接口、编译基础设施、语义 gate、性能剖析 feedback 和 evidence memory 才能可靠服务软硬件协同。
13. 第一章 1.3 必须明确区分架构接口和编译接口。架构接口定义硬件可提供的能力边界；编译接口定义软件如何发现、使用、优化和验证这些能力。
14. 第一章 1.3 末尾或 1.5 开头需要一个中心论断句，用来从背景综述过渡到三个核心研究问题。具体措辞在正文写作时润色。
15. 第一章必须使用业界数据、顶级体系结构会议论文或权威系统论文支撑关键判断。证据优先来自大公司作者机构、工业研究实验室、权威专家、工业系统论文和官方技术报告。写作前阅读 `docs/writing/introduction-evidence-plan.md`，并为每个数据点核验指标、基线、工作负载和适用范围。
16. 第一章证据写法优先采用“关键转折处代表性例子”。证据表格可选，不是硬性要求；只有在增强可核验性且不破坏学术叙事流畅性时才放入正文。
17. 第一章应吸收“编译基础设施作为 agent harness 介质”的论断：编译基础设施不是后端工程附属品，而是组织目标事实、IR、语义契约、受约束变换、可执行产物、profiling/trace evidence 和 gate 的中间层。参考材料为 `resources/compiler-infra-is-harness-medium.md`，但正文必须写成中文学术论证。
18. 支撑 agentic/LLM 趋势时，优先使用 NVIDIA AVO、Google DeepMind AlphaEvolve、Anthropic/OpenAI 官方技术报告或系统说明；其次使用 Siemens/Synopsys 的 EDA agentic 产品/路线材料、METR 长程任务能力模型、Agentic Software Engineering / Agentic EDA / CompilerGPT / AwareCompiler 等前沿论文。必须标明来源性质：AVO/AlphaEvolve 用于技术机制和可验证 evaluator/harness，Anthropic/OpenAI 用于 agentic coding、tool use、sandbox/test/review/human oversight/safety boundary，厂商页面用于行业动向，论文用于研究问题和技术挑战。
19. 第一章 1.4 的智能体化方法小类应形成递进链条：OpenAI/Anthropic 工程 agent -> AlphaEvolve evaluator-constrained coding agent -> NVIDIA AVO 工业 GPU kernel/compiler optimization -> Agentic EDA/verification-agent 风险 -> 本报告的模型-工具-验证约束体系判断。
20. 写 AlphaEvolve 与 EggMind 的关系时只做轻量对照：共同点是都通过可执行或可验证反馈约束搜索；区别是 EggMind 更贴近编译优化与等价空间，能更自然过渡到 IntelliC 的 compiler-native harness。
21. 写 AVO 与第五章 5.4 的关系时，AVO 是方向证据，不是窄义竞品。必须说明本报告拟研究的长程芯片算子自动优化 Harness 具有更宽对象、更系统约束、更异构硬件假设和更长期的可维护证据记忆。
22. 每个技术方向必须回答三个问题：
   - 为什么行业需要这个方向？
   - 现有方法卡在哪里？
   - 本人的工作如何构成可复用的方法、工具或系统能力？
23. 避免成果堆砌。每项工作都要放回主线中，说明它在“系统生产力”链条中的位置。
24. 第一作者代表性工作要详尽展开：问题背景、关键挑战、核心思想、系统设计、技术机制、实验设置、结果和意义都要写。
25. 其他工作概括介绍：强调它们如何补全技术链条，不展开实现细节。
26. 未来工作不能变成项目列表；必须从已有工作自然推出，说明未来要解决的剩余矛盾。
27. 第一层“面向领域定制的架构接口与端到端协同”中，顺序固定为 APS/Aquas -> ISAMORE -> Cayman。APS/Aquas 作为框架入口先写，ISAMORE 回答“定制什么”，Cayman 讨论微架构约束。ISAMORE 是同等重要的代表性工作，正文篇幅不得少于 APS/Aquas 的合计篇幅。
28. 协同框架和单点优化技术同样重要。不要为了强调系统性而压缩单点优化技术的技术深度。
29. 当前正文只能视为骨架。后续质量提升必须显著扩写各部分，形成耐心、严肃、完整的故事讲述。

## 语言风格

1. 中文为主，保留必要英文术语，如 MLIR、e-graph、equality saturation、anti-unification、ISAX、ASIP、RTL、HLS、LLM、TraceDB。
2. 中文句子要短，避免一段中连续堆叠三个以上抽象名词。
3. 技术术语第一次出现时给出中文解释或上下文。
4. 不使用宣传式表达，例如“国际领先”“填补空白”等，除非有明确证据。优先写“提出了”“实现了”“验证了”“支撑了”。
5. 结果数字必须带比较对象和场景，例如“相比 Vitis HLS 平均 3.01x、最高 5.22x”。

## 结构规范

建议正文结构：

1. 绪论：算力紧缺、领域定制、芯片设计生产力瓶颈。
2. 国内外研究现状与挑战：敏捷芯片设计、ASIP/ISAX、硬件设计前端、HLS/综合、AI for EDA。
3. 研究基础与已有工作：
   - 面向领域定制的架构接口与端到端协同：APS/Aquas + ISAMORE + Cayman。
   - 面向高质量硬件实现的前端抽象与综合优化：HECTOR、Cement、Clay、SkyEgg。HECTOR 优先提出为早期技术框架积累，但只着重概括；Clay/SkyEgg 可说明其对 HECTOR 技术框架的继承或依赖。
   - Cement 和 Clay 写成分工关系，不写成版本递进：Cement 偏硬件行为表达和周期确定性，Clay 偏实现质量与综合机制。
   - SkyEgg 写成本章方法高潮和通向第四章 EggMind/策略智能体化的过渡点，但不能压缩 Clay 的重要篇幅。
   - 大模型与形式化技术驱动的软硬件协同方法：EggMind 是绝对重点，按完整现有工作展开；OriGen 像 HECTOR 一样作为所在章节的重要背景，不作为主要章节或代表性工作展开，只用于说明大模型硬件生成的数据、反馈修复和验证约束背景，并连接未来“验证约束下的架构、硬件与编译协同生成”方向。
4. 未来研究计划：
   - 验证约束下的架构、硬件与编译协同生成。
   - 智能体编译基础设施。
   - 面向工业级异构架构的算子优化与智能体化编译协同。
   - IntelliC、Spine、PTO Runtime distributed features 只能作为已有基础、原型或内部代号出现，不作为方向标题。
   - Spine 不能只承接 OriGen；它也必须承接 APS/Aquas 的端到端协同框架，是 APS/Aquas 思想的 agentic 化延伸，并将应用意图、架构接口、硬件生成、编译 lowering、执行 oracle 和跨层检查纳入验证约束。
   - IntelliC/编译基础设施应被写成 agentic retargeting 的 harness medium：desired programming face + target facts -> generated compiler middle -> backend artifacts。它与 Spine 的关系是：Spine 提供架构/硬件描述和证据，IntelliC 提供编译生成、验证和证据组织层。
   - `resources/seed-proposal.docx` 的内容属于“面向工业级异构架构的算子优化与智能体化编译协同”方向，写作时应抽象为研究问题和技术路线，不照搬项目申请书格式。
   - Agent Harness 只在长程芯片算子自动优化等需要它的子线中出现；不要把 PTO Runtime distributed features 写成 agent 项目。
   - 算子自动优化 Agent Harness 与 PTO Runtime distributed features 是并列子线，汇合点是工业级异构架构；不要强行写成上下游闭环。
5. 预期成果与计划安排。

## 图示规范

1. 正文需要适当添加图片，支撑总体逻辑、系统结构、方法流程和实验结果说明。
2. 已有系统图、流程图、实验图优先从 `resources/` 中选用，使用前必须确认图的来源、含义和与正文论点的对应关系。
3. 如果 `resources/` 中没有合适图片，例如整体研究主线、章节关系、未来工作闭环、跨层软硬件协同框架等，应直接绘制新图。
4. 第一章必须新绘制两张总览图：一张“跨硬件架构软硬件协同生态”概念图，一张“研究问题与章节映射”结构图。
5. 新图必须服务论证，不做装饰。优先使用框架图、流程图、分层图和验证约束体系图。
6. 每张图都要在正文中被解释：图前说明它回答的问题，图后说明它支撑的论点。
7. 图题和图内文字中文为主，保留必要英文术语，并与术语表保持一致。

## 代表性工作展开模板

每个代表性工作按以下顺序写：

1. 背景：该问题对应哪类产业或学术瓶颈。
2. 挑战：至少两点，且要具体。
3. 方法：一句话概括核心洞见，再分解为系统组件。
4. 结果：性能、资源、开源、会议、落地等可验证指标。
5. 意义：它如何补强主线，而不只是单点优化。

ISAMORE 展开时必须同时详细回答：

1. 可复用自定义指令为什么是敏捷芯片设计中连接应用模式与硬件能力的关键机制。
2. e-graph anti-unification 为什么是发现可复用指令模式的核心方法创新。

## 迭代规则

1. 每次迭代先检查主线是否一致，再改局部文字。
2. 当新增成果时，先更新 `report_spec.md` 的成果矩阵，再更新正文。
3. 当无法确认事实时，在正文用 `TODO:` 标记，不把猜测写成定论。
4. 正式提交前至少完成三轮检查：
   - 逻辑闭环检查：背景、挑战、方案、未来是否呼应。
   - 事实检查：数字、会议、作者顺序、专利、获奖是否可追溯。
   - 语言检查：中文是否顺畅，术语是否统一，段落是否过长。
# 图表与组图

- 引用源文图表前，先确认该图在源文 LaTeX 中的实际使用方式。
- 来自源文或本地资料的图片必须以 PDF 格式放入 `report/assets/` 并由正文引用；不要直接引用 PNG、JPG、SVG 等格式。GPT 自动绘制的新图除外。
- 源文组图必须作为一个完整语义单元保留，不得只插入其中某个子图。
- 如果只有子图资产可用，用 Typst `grid` 或 `stack` 在一个父 `#figure(...)` 中重组完整组图；只给父图正式 caption，子图用 `(a)`、`(b)` 局部说明。
- 如果正文只需要讨论某个 panel，在正文或图题中说明对应 panel，不裁剪或单独插入该 panel。
